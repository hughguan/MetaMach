
### ── 面向新任厂长的硅基工业级生产调度中枢业务指南

> **厂长视点 (Director's Note)**
> 
> 本规范书（Spec）专为新任“厂长”（业务最终用户）打造。作为厂长，您无需理解 Rust、UDS Socket 或 PostgreSQL 的底层代码。您的核心职责是：**注册新产品（Blueprint）、调度 SOP 工作流（Workflow）、审批高危操作（HITL Gate）以及查看产品最终交付的质检报告（Production Report）**。本文档将从业务视角为您全面拆解 MetaMach 2.0 的功能版图。

## 1. 业务愿景与核心痛点

在传统的 AI 辅助研发与自动化装配中，厂长经常面临以下三大痛点：

1. **黑盒不可控（安全红线）** AI Agent 犹如脱缰野马，在黑盒终端里乱写代码和指令，可能直接删库或引发物理硬件引脚冲突，缺乏安全防护和人工确认门禁。
    
2. **脆弱不耐久（运行红线）** 网络稍微抖动、SSH 断开或电脑休眠，跑了数小时的重型编译和部署任务便瞬间崩溃，无法恢复，前功尽弃。
    
3. **知识不沉淀（进化红线）** AI 每次遇到的报错、引脚冲突和修复经验，随着会话结束直接消失，无法转化为工厂的常驻资产，导致下次开发时重复踩坑。
    

**MetaMach 2.0** 通过“常驻守护大脑 + 耐久物理会话 + 共享脑图”的设计，提供像物理实体工厂一样确定、安全、可控且不断自我进化的数字化硅基流水线。

## 2. 厂长核心业务功能模块

### 2.1 产品蓝图管理 (Blueprint Onboarding & Offboarding)

产品蓝图（Blueprint）是工厂生产的最小业务单元（例如：`joyrobots` 教育机器人或 `gatemetric` BMX 姿态评估系统）。

- **蓝图一键上线 (Onboard)：** 厂长在 Day 0 将一个新产品线引入车间的唯一入口。厂长只需执行 `janus onboard --blueprint <name>`（或通过 Popup 控制台选择“上线新产品”），系统即按以下标准上线工艺自动接管：
    
    1. **配方校验：** 读取 `blueprints/<name>/janus.toml`，校验产品名、默认流水线（`default_workflow`）、远程 SSH 靶机、OpenWiki 脑图索引范围；并确认 `workflows/<default_workflow>.toml` 工艺文件真实存在。
        
    2. **点火前自检：** 探测 Unified DB 可达、tmux 引擎就位；针对跨主机蓝图（如 `gatemetric`）尽力探测远程编译靶机的 SSH 可达性（不可达仅告警，不阻断上线）。
        
    3. **租户注册：** 在 Unified DB 中为该蓝图分配独立的逻辑租户隔离空间（以 `blueprint_id` 为分区键），写入一行 `ACTIVE` 状态的蓝图元数据。该操作**幂等**——重复执行不会产生脏数据，重新上线一个已下线的蓝图会将其由 `OFFBOARDED` 重新激活。
        
    4. **流水线绑定：** 绑定 `janus.toml` 声明的默认 SOP 工作流，使其立即可被派发。
        
    5. **脑图载入与经验遗传：** 加载该产品专属的 `blueprints/<name>/openwiki/` 知识脑图。**若该蓝图历史上曾下线并生成过 `production_report.md`**，系统优先索引该质检白皮书，并将其中关键的引脚冲突、编译报错等避坑经验以 `## Previous Incidents` 少样本（Few-shot）形式注入新一代 Agent 的 System Prompt，实现硅基经验的跨代遗传。
        
    6. **上线就绪：** 状态置为 `ACTIVE`，产品线即时出现在 Popup 派单菜单中，厂长可立即派发流水线。
        
- **蓝图无损下线 (Offboard)：** 当产品研发完毕需要封线时，厂长执行 `janus offboard --blueprint <name>`。系统自动提取该项目开发期间的所有 Steps 运行轨迹，通过大模型熔炼生成一份最新的“质检报告与生产白皮书 (production_report.md)”沉淀入脑图，同时彻底清空数据库中过期的冗余缓存，实现数据库无损瘦身。下一次 Onboard 该蓝图时，这份白皮书将被自动回收为新一代 Agent 的免疫抗体（见上文“脑图载入与经验遗传”）。
    

### 2.2 SOP 工艺流水线调度 (Workflow Dispatcher)

流水线是标准化、声明式的工艺步骤集合。厂长无需手动干预每一个步骤，只需整体调度：

- **可视化排班菜单：** 厂长通过 Herdr 终端按下快捷键 `prefix+j`，会在屏幕中央弹出一个优雅的 **Popup 控制面板**。厂长可在此查看所有可用产品，并一键分派标准 SOP（如“研发装配线”、“诊断除错线”或“固件跨端烧录线”）。
    
- **跨物理主机协同：** 流水线支持多工位协作。例如，Step 1 在本地拉起 AI 编写滤波算法；Step 2 自动通过安全通道将代码投射至远端的重型交叉编译服务器上，全程无需厂长手动切终端、配置 SSH。
    

### 2.3 物理会话“长生不老”保障 (Tether Engine)

确保任何研发任务在物理层面的强健性：

- **断线不挂机 (Remain-on-Exit)：** 所有底层任务（无论是本地编译还是 SSH 远端烧录）均被锁死在独立的物理 Session 中。即便厂长拔掉网线、关闭终端甚至电脑死机，任务依然在服务器后台平稳运行。
    
- **现场保留与秒级挂接：** 当任务运行完毕或中途报错时，现场 100% 留存。厂长随时可以一键“Attach（挂接）”进入报错终端的物理现场，人工调优后直接无缝恢复。
    

### 2.4 智能安全警卫与人工审批门禁 (Tool Guard & HITL Gate)

MetaMach 2.0 奉行“安全第一”的原则，将 AI 关在安全的制度笼子里：

- **高危指令主动拦截：** 当 AI 试图执行超出其资质等级的操作（如修改关键系统配置文件、执行高风险删除指令或检测到硬件配置冲突）时，底层代理 Shell（`janus-sh`）会在内核级将其截获并强制挂起任务。
    
- **多端合闸审批 (Teams / TUI)：** 安全门禁触发后，厂长的 **MS Teams / Telegram 手机端**会瞬间弹出一张带有报错现场上下文的卡片（包含原因、受影响模块及 `[一键 Resume]` / `[拒绝执行]` 按钮）。厂长在移动端点击按钮即可远程控制车间装配线的启停。
    

### 2.5 工作流进度大盘 (Workflow Monitor)

派单之后，厂长需要一个“车间生产大屏”来回答三个核心问题：**它还在跑吗？现在卡在第几个工位？是正常运行还是卡死了？**

- **实时进度大盘：** 厂长按下 `prefix+j` 唤醒 Popup 后，可在“派单 (Dispatch)”与“进度 (Progress)”两个视图间一键切换。进度视图以矩阵形式罗列所有在途工作流：所属蓝图、流水线名称、当前工位（Step）、每个工位的状态（`PENDING` / `RUNNING` / `COMPLETED` / `SUSPENDED` / `FAILED`）、已耗时，以及最近一段被截断的终端输出（Stdout 摘要）。大盘每 1–2 秒自动刷新。
    
- **挂起即时高亮：** 当某个工位触发安全熔断进入 `SUSPENDED` 时，该行在 1 秒内被高亮标红，并直接附带 `[Attach 现场]` / `[一键 Resume]` 入口，厂长无需在多个终端间来回切换寻找卡死的任务。
    
- **多蓝图并列运营：** 当车间同时有多个蓝图在产（如 `joyrobots` 与 `gatemetric` 并行），大盘按蓝图分组独立展示各自的工作流进度，互不干扰。
    
- **无界面环境兜底：** 在仅可 SSH 的无 TUI 环境（如 CI 或远程服务器），厂长可执行 `janus status` 获取同等信息的纯文本/JSON 快照，秒级掌握车间全局。

## 4. 厂长一日业务时序场景 (User Journey)

为便于您快速代入，以下是厂长使用 MetaMach 2.0 进行日常生产调度的典型交互流程：

### 🆕 Day 0：上线一个全新蓝图 (Onboarding)

> 厂长面对的是一台刚通电、零产品线的空车间。第一件事不是派单，而是把新产品线“上线”。

1. **准备配方：** 厂长在 `blueprints/gatemetric/` 下放好产品源码与一份 `janus.toml`（声明默认流水线 `firmware-deploy`、远程 SSH 编译靶机、OpenWiki 脑图索引范围）。
    
2. **一键上线：** 厂长执行 `janus onboard --blueprint gatemetric`（或在 Popup 中选择“上线新产品”）。
    
3. **系统自动接管：** 系统校验配方 → 自检 DB 与 tmux → 在 Unified DB 注册 `ACTIVE` 租户 → 绑定 `firmware-deploy` 流水线 → 载入专属脑图（若存在上一代的 `production_report.md`，自动回收为免疫抗体）。
    
4. **即时可用：** 终端打印上线成功，`gatemetric` 随即出现在 Popup 派单菜单中。至此车间从零产品进入可生产状态。

### 🎯 流程实操：GateMetric BMX 姿态评估板研发

1. **上午 09:00 - 派发工单：** 厂长进入 Richmond Hill 车间，打开终端，按下 `prefix+j`。弹窗中选择产品 `gatemetric` 并派发 `dev-flow`（研发装配 SOP）。
    
2. **上午 09:05 - 巡盘查进度：** 派单后厂长并未盲等，而是在 Popup 中切换到“进度”视图。大盘显示 `dev-flow` 已进入 Step 1（Scout 扫描，`RUNNING`），并实时滚动最近一段终端输出。厂长确认流水线正常推进后离开工位。
    
3. **上午 09:15 - AI 自动装配与安全拦截：** AI 进场，在本地沙箱编写 ESP32 的滤波算法。AI 试图通过命令行直接修改主板的核心引脚配置以调试外设。此时 `janus-sh` 安全警卫拉响警报：_“检测到引脚冲突，有物理烧板风险！”_。任务被自动非破坏性挂起，本地及远程物理会话保持不灭。进度大盘上该工位在 1 秒内变红高亮为 `SUSPENDED`，并直接弹出 `[Attach 现场]` / `[一键 Resume]` 入口。
    
4. **上午 09:16 - 多端合闸（两条现实路径，不再自相矛盾）：** 进度大盘高亮 `SUSPENDED` 的同时，厂长手机 Teams 收到报警卡片。视厂长所处场景走其中一条路径：
    
    - **路径 A（移动端 ── 厂长正在开会，身边只有手机）：** 厂长在手机上阅读报错上下文，点击 **`[标记待人工修复]`**。系统回复“等待厂长回终端就地修复”，工位保持 `SUSPENDED`、物理会话不灭。会议结束后厂长回到工位，`attach` 进报错窗格就地改写 C++ 头文件的引脚定义，键入 `metamach-resume` 发出完成信号，Daemon 派发下一工位。
        
    - **路径 B（桌面端 ── 厂长就在终端前）：** 厂长直接在 TUI 收到同一张卡片，立即 `attach` 进报错窗格就地修复，键入 `metamach-resume`（或点击 **`[🔄 Resume]`**）恢复，流水线无感接棒。
        
    两条路径均**不盲目重跑被拦截的原命令**，而是就地修复后派发下一工位，避免覆盖人工修复。
    
5. **下午 14:00 - 自动进化与封线：** 交叉编译和远程部署烧录顺利通过。厂长在控制台执行 `janus offboard --blueprint gatemetric`。系统瞬间抹除本地数据库缓存的大字段以防爆库，并在 OpenWiki 目录下自动沉淀生成一份 `production_report.md`，记录了本次引脚冲突及解决方案。
    
6. **未来：** 下一次厂长对 `gatemetric` 执行 `janus onboard` 重新上线时，系统优先索引该 `production_report.md`，并将引脚冲突等避坑经验以 `## Previous Incidents` 少样本注入新一代 Agent 的 System Prompt，使其进场即免疫同类错误。
    

## 5. 交付物标准与质量红线

- **数据 Footprints 绝对防爆：** 任何步骤日志抓取强行限制在 **16KB 预算** 内。超过部分在入库前截断，且下线（Offboard）后大 JSON 必须彻底物理擦除，防止数据库因体积过大产生“熔断崩溃”。
    
- **零依赖开箱即用：** 一键运行 `make bootstrap` 应自动拉起 PG 数据库容器，完成软链接配制，对厂长完全屏蔽复杂的底层基础设施配置。
