# M4 Task 4.1 Remaining Work - Design Plan (cross-host SSH + cold-start re-exec)

> **Status:** Design rev 3 (option b - absurd adapter). **Phase 0a (absurd adapter) + Phase 0b (workflow engine) are implemented** (commits `64c8f11` + this PR) - `Request::Dispatch` (Contract 3.11), `workflow::run_workflow`, `tmux_alive` wired, `utc_03_01b` green. Phase 1 (cold-start re-exec) + Phase 2 (cross-host SSH) + Task 4.4 still deferred.
> **Scope:** The last two-thirds of M4 Task 4.1: cross-host SSH transport + checkpoint-driven cold-start re-exec.
> **Direction (rev 3):** the workflow-execution engine is built on **absurd** (`janus/sql/absurd.sql`, vendored v0.4.0) - build the Rust adapter (`init_absurd_schema` + `AbsurdPgAdapter` per `docs/Absurd-Integration.md` §1/§7.3) and use absurd's `spawn_task`/`claim_task`/`complete_run`/`set_task_checkpoint_state`/`await_event`/`emit_event`/`create_queue`. This supersedes rev 2's "MetaMach-owned `metamach_tasks`" (option a) and aligns with the Absurd-Integration doc. `metamach_step_meta` (002) stays as MetaMach's thin overlay for fields absurd has no concept of (`target_sha`, `stdout_tail`, `hitl_verdict`, `session_name`).
> **Prerequisite finding:** Both named items are blocked on the **workflow-execution engine**, which does not exist. The daemon today handles only `Ping`/`Blueprints`/`Progress`/`GuardCheck`/`Onboard`/`Offboard` - no `Dispatch`/`RunStep`, no step-state machine, nothing drives `janus::tmux`. `coldstart::reconcile` only logs (nothing dispatches). `absurd.sql` is vendored but **not yet loaded** (no adapter).

This plan covers **three phases**: the workflow-execution engine on absurd (Phase 0, the prerequisite), cold-start re-exec (Phase 1), and cross-host SSH transport (Phase 2).

## Phase 0 - Workflow-execution engine on absurd (prerequisite)

### 0.1 Absurd adapter (load the schema) - `docs/Absurd-Integration.md` §1

`absurd.sql` is vendored but never executed. Build the adapter:

```rust

    let absurd_sql = include_str!("../sql/absurd.sql");

    verify_absurd_schema_version(pool).await?;  // §7.1: EXPECTED_ABSURD_VERSION guard
    Ok(())
}
```

Wire it into `ensure_blueprint_db` **before** 002/003 - so the `002_blueprint.sql` header "Applied on `janus onboard` AFTER `absurdctl init`" becomes literally true (the adapter IS the init). absurd installs its `absurd` schema + the per-queue `t_`/`r_`/`c_`/`e_`/`w_`/`i_` tables; 002 adds MetaMach's `metamach_step_meta` overlay on top.

**No `metamach_tasks` table** (rev 2's option a is dropped) - absurd owns task/run/checkpoint state. `metamach_step_meta` (002) is extended with one column for tmux wiring (review finding #4):

```sql
-- 002_blueprint.sql: add to metamach_step_meta
ALTER TABLE metamach_step_meta ADD COLUMN IF NOT EXISTS session_name VARCHAR(100);
-- target_sha, exit_code, stdout_tail, started_at, hitl_verdict already exist.
```

`session_name` is MetaMach-only (absurd doesn't know about tmux). `dispatch_sha` is not needed as a separate task-level column - cold-start race detection compares the step's `target_sha` against current HEAD on resume (see §1.2).

### 0.2 `AbsurdPgAdapter` - `docs/Absurd-Integration.md` §7.3

A MetaMach-defined trait decoupling the engine from the SQL contract:

```rust
// janus/src/absurd/adapter.rs (new)
#[async_trait] pub trait DurableEngine: Send + Sync {
    async fn create_queue(&self, name: &str) -> Result<()>;
    async fn spawn_task(&self, queue: &str, task_id: Uuid, payload: &serde_json::Value) -> Result<()>;
    async fn claim_task(&self, queue: &str) -> Result<Option<ClaimedTask>>;
    async fn complete_run(&self, queue: &str, run_id: Uuid, result: &serde_json::Value) -> Result<()>;
    async fn fail_run(&self, queue: &str, run_id: Uuid, err: &str) -> Result<()>;
    async fn set_checkpoint(&self, queue: &str, task_id: Uuid, step: &str, state: &serde_json::Value) -> Result<()>;
    async fn get_last_checkpoint(&self, queue: &str, task_id: Uuid) -> Result<Option<(String, serde_json::Value)>>;
    async fn await_event(&self, queue: &str, name: &str) -> Result<Event>;       // HITL SUSPEND
    async fn emit_event(&self, queue: &str, name: &str, payload: &serde_json::Value) -> Result<()>;
    async fn non_terminal_tasks(&self, queue: &str) -> Result<Vec<TaskInfo>>;    // cold-start
}
pub struct AbsurdPgAdapter { pool: PgPool }  // SELECT absurd.spawn_task(...) etc.
```

~100 lines of sqlx queries calling absurd's stored procs. A `FakeEngine` test impl mirrors `FakeBackend`.

### 0.4 The engine: `janus/src/workflow.rs` (new)

`pub async fn run_workflow(db, engine: &dyn DurableEngine, backend: &dyn DurableBackend, recipe, workflow, repo_root) -> Result<Uuid>`:

1. `engine.create_queue("<blueprint>.<workflow>")` (idempotent - onboard can also do this). Mint `task_id = Uuid::new_v4()`; `engine.spawn_task(queue, task_id, payload)` (payload = step list).
2. Resolve steps from the `Workflow` (`recipe::load_workflow`). For each step in order:
   - Upsert `metamach_step_meta` (`status`=`PENDING`->`STARTING`->`RUNNING`); pin `target_sha` = current git HEAD at `repo_root` (all-zeros for non-git).
   - `engine.claim_task(queue)` (pull-mode lease); build the tmux session `SessionId::new_for_task(task_id)`; record `step.session_name`.
   - The step runs under `janush` as `SHELL` (confirmed - review #10: interception is the point). Step `command` -> `janush -c "<command>"`; set `JANUS_AGENT=<step.agent>` in the session env. `cwd` = `repo_root`.
   - `backend.create_session(id, command, cwd)`. Poll for exit. **Exit-code capture (review #5):** `tmux display-message -p -t <id> '#{pane_dead_status}'` (tmux 3.3+, already mandated). `backend.capture_pane(id)` -> `stdout_tail` (16 KiB via `truncate_16k`).
   - On exit 0: `engine.complete_run(...)` + `engine.set_checkpoint(queue, task_id, step, state)` + `step.status`=`COMPLETED`. Non-zero: `engine.fail_run(...)` + `step.status`=`FAILED`, task stops.
   - On a Tool-Guard `BLOCK` (require_approval): `engine.await_event("hitl.approve:<run_id>")` (task SUSPENDs in PG, zero CPU) + `step.status`=`SUSPENDED` + `step.hitl_verdict`=NULL + gateway dispatches the HITL card. **Stop** (resume via `emit_event` - §4 follow-on).
3. All steps `COMPLETED` -> task done.

**State machine** (matches CLAUDE.md / absurd's run states):
- Task: `PENDING -> STARTING -> RUNNING -> COMPLETED | FAILED | SUSPENDED`
- Step: `PENDING -> STARTING -> RUNNING -> COMPLETED | FAILED | SUSPENDED`

### 0.5 `tmux_alive` wiring (Contract 3.3)

`ActiveTask.tmux_alive` ("lands with Task 2.4" today) wired: `db.progress(...)` reads `metamach_step_meta.session_name` and calls `backend.has_session(session_name)`. The daemon holds a `DurableBackend` handle for this second-pass liveness check.

## Phase 1 - Cold-start re-exec (named Task 4.1 work)

### 1.1 Make `coldstart::reconcile` actually resume

Today it logs only. Change it to: for each `engine.non_terminal_tasks(queue)` (absurd's task table - one row per run, review #3):
1. Read the last `COMPLETED` checkpoint via `engine.get_last_checkpoint(queue, task_id)`.
2. Determine the next step (after the last `COMPLETED`; or re-run the interrupted step if it was mid-`RUNNING` and its tmux session is gone).
3. Mint a fresh `tmux-janus-task-<task_id>-<new_uuid>` session (ARCH §6.1 - never reuse stale names).
4. Call `workflow::resume_from(task_id, next_step)` - same as `run_workflow` but starts at a step index, skipping `COMPLETED` steps. Same `task_id` (preserves audit trail).
5. Reject `tmux-resurrect` (already the case - reads from PG/absurd checkpoints, never tmux state).

### 1.2 Race detection on resume (Task 4.4 tie-in)
On resume, compare the step's `target_sha` (pinned at dispatch) against current HEAD. Mismatch => the step was dispatched against stale code; mark `SUSPENDED` + `CONCURRENCY_RACE_ALERT` + auto-reschedule a new task against the new HEAD (Task 4.4 - the `target_sha` column already exists on `metamach_step_meta`).

### 1.3 Tests (Phase 1)
- **PG + tmux integration (ignored):** `utc_03_03` (real) - dispatch a 2-step workflow, kill the daemon mid-step-2, restart PG + daemon (NOT `make bootstrap`), assert `reconcile` resumes from the checkpoint and reaches `COMPLETED`.

### 1.4 Mid-execution retry loop (`max_attempts: 3`)

Phase 0b ships `max_attempts: 1` (no auto-retry): the engine is a one-shot worker (claim -> run -> complete/fail -> return) that does NOT re-claim, so a retry run absurd schedules after `fail_run` would orphan (`claim_task` doesn't long-poll; `coldstart::reconcile` only runs at daemon start and only resumes non-terminal tasks - a `FAILED` task is terminal). So a transient failure (e.g. cargo build timeout on a slow network) permanently fails the task until manual re-dispatch.

Phase 1 closes this by making `run_workflow` a **retry-claim loop** + flipping `spawn_task` to `max_attempts: 3`:
1. After `fail_run`, loop: re-`claim_task` (polling through absurd's backoff - claim returns `None` while the retry run isn't available yet; sleep + retry).
2. Distinguish "retry pending" from "terminal": if `claim_task` returns `None`, check `engine.non_terminal_tasks(queue)` - if the task is still non-terminal, a retry is pending (keep polling); if terminal, stop.
3. Resume from the last `COMPLETED` checkpoint via `get_last_checkpoint` (skip done steps, re-run the failed one) - same `resume_from` logic as §1.1, just driven mid-session instead of at cold start.
4. Stop when the task is terminal (`completed` after success, `failed` after `max_attempts` exhausted).

`FakeEngine` must then model absurd's retry (a `max_attempts` counter per task; `fail_run` creates a new claimable run until exhausted) so the unit test can assert the retry + resume path.

## Phase 2 - Cross-host SSH transport (named Task 4.1 work)

### 2.1 `SshTmuxBackend` (`janus/src/tmux/ssh.rs`, new)

Second `DurableBackend` impl driving tmux on a remote host over SSH:

```rust
pub struct SshTmuxBackend { host: String, user: Option<String> }
impl DurableBackend for SshTmuxBackend { ... }  // ssh [-l user] host tmux -L metamach-tmux ...
```

- `create_session`: `ssh <host> tmux -L metamach-tmux new-session -d -s <id> -c <cwd> '<command>'` with `remain-on-exit on`.
- `capture_pane`/`has_session`/`kill_session`/`list_sessions`: same `ssh ... tmux ...` prefix.
- `display-message '#{pane_dead_status}'` works identically over SSH.
- SSH flags: `-o BatchMode=yes -o ConnectTimeout=5 -o StrictHostKeyChecking=accept-new` (key-based only, no password/agent-forwarding - Deployment-Spec §4.2, review #9).

### 2.2 Backend selection per step

`WorkflowStep.host` -> blueprint `[remote]` fallback. `None`/`"local"` => `TmuxBackend`; set => `SshTmuxBackend`. The engine selects per step.

### 2.3 Tests (Phase 2)
- **Unit:** `SshTmuxBackend` argv construction (no real SSH).
- **SSH-gated (ignored, `requires SSH credentials`):** `utc_03_02` - dispatch a step to a real SSH compile host; assert the remote session is created and survives a packet-drop/re-attach. CI: separate `continue-on-error` step (PR #2 forward-compatible comment).
