# M4 Task 4.1 Remaining Work — Design Plan (cross-host SSH + cold-start re-exec)

> **Status:** Design for review — no implementation yet. Code lands in a later session.
> **Scope:** The last two-thirds of M4 Task 4.1: cross-host SSH transport + checkpoint-driven cold-start re-exec.
> **Prerequisite finding:** Both are blocked on the **workflow-execution engine**, which does not exist. The daemon today handles only `Ping`/`Blueprints`/`Progress`/`GuardCheck`/`Onboard`/`Offboard` — there is no `Dispatch`/`RunStep` request, no `spawn_task`, no step-state machine, and nothing drives a workflow's steps into `janus::tmux`. `janus::tmux` has session management (`DurableBackend`: create/attach/kill/has/capture) but **nothing calls it**. `coldstart::reconcile` scans for non-terminal tasks but finds none (nothing dispatches), so it only logs. `absurd` is branding (in-repo sqlx pool), not the external engine the comments reference.

This plan therefore covers **three phases**: the workflow-execution engine (Phase 0, the prerequisite), cold-start re-exec (Phase 1), and cross-host SSH transport (Phase 2). Phases 1 and 2 are the named Task 4.1 work; Phase 0 is the unblocker.

---

## Phase 0 — Workflow-execution engine (prerequisite)

### 0.1 Schema: a task table

The per-blueprint DB today has only `metamach_step_meta` (per-step rows keyed by `task_id` + `step_name`). There is **no task table** — a workflow run has no overall status row. The 002 comment assumed absurd's `absurd_tasks` table would provide this, but absurd isn't integrated.

**Decision:** add a MetaMach-owned `metamach_tasks` table (migration `004_tasks.sql`), applied alongside 002/003 in `ensure_blueprint_db`.

```sql
CREATE TABLE IF NOT EXISTS metamach_tasks (
    task_id          UUID PRIMARY KEY,
    blueprint_name   VARCHAR(100) NOT NULL,
    workflow_name    VARCHAR(100) NOT NULL,
    status           VARCHAR(20)  NOT NULL DEFAULT 'PENDING',  -- PENDING|STARTING|RUNNING|COMPLETED|FAILED|SUSPENDED
    current_step     VARCHAR(100),
    suspended_reason TEXT,
    started_at       TIMESTAMPTZ,
    ended_at         TIMESTAMPTZ,
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_tasks_status ON metamach_tasks(status);
```

`metamach_step_meta` already carries per-step `status`/`exit_code`/`stdout_tail`/`started_at`/`hitl_verdict` — reused as-is.

### 0.2 Protocol: `Dispatch` contract

Add to `janus/src/protocol.rs`:

```rust
pub enum Request {
    // ...existing...
    Dispatch { blueprint: String, workflow: Option<String> },  // None => recipe.default_workflow
}

pub enum Response {
    // ...existing...
    Dispatch { task_id: Uuid },
}
```

Wire `Request::Dispatch` in the daemon handler (`janus_daemon.rs`) → calls `workflow::run_workflow(...)`. Document as a new Contract (e.g. 3.9 Dispatch) in `Feature-Spec.md`.

### 0.3 The engine: `janus/src/workflow.rs` (new module)

`pub async fn run_workflow(db, backend: &dyn DurableBackend, recipe: &ValidatedRecipe, workflow: &Workflow, repo_root) -> Result<Uuid>`:

1. Insert a `metamach_tasks` row (status=`STARTING`), mint `task_id = Uuid::new_v4()`.
2. Resolve steps from the `Workflow`. For each step in order:
   - Upsert `metamach_step_meta` (status=`PENDING` → `STARTING` → `RUNNING`); set `task.current_step`.
   - Set task status=`RUNNING`.
   - Build the tmux session: `SessionId::new_for_task(task_id)`; command = the step's `command` (run under `janush` as `SHELL` so Tool Guard intercepts — reuses the existing proxy-shell path). `cwd` = `repo_root`.
   - `backend.create_session(id, command, cwd)`.
   - Poll `backend` for session exit (`remain-on-exit` keeps the pane; capture exit code via the pane's shell exit). On exit: `backend.capture_pane(id)` → `stdout_tail` (16 KiB budget via `truncate_16k`); set `step.exit_code`, `step.status` = `COMPLETED` (exit 0) | `FAILED` (non-zero).
   - On a Tool-Guard `BLOCK` (require_approval) verdict mid-step: set `step.status`=`SUSPENDED`, `step.hitl_verdict`=NULL (pending), `task.status`=`SUSPENDED`, `task.suspended_reason`, and **stop** (await HITL resume — the resume/continue path is the M4 HITL loop, partly present via the gateway).
3. If all steps `COMPLETED`: `task.status`=`COMPLETED`, `ended_at=NOW()`. On any `FAILED`: `task.status`=`FAILED`.

**State machine** (matches the CLAUDE.md status enum):
- Task: `PENDING → STARTING → RUNNING → COMPLETED | FAILED | SUSPENDED`
- Step: `PENDING → STARTING → RUNNING → COMPLETED | FAILED | SUSPENDED`

### 0.4 `tmux_alive` wiring (Contract 3.3)

`ActiveTask.tmux_alive` is documented as "lands with Task 2.4 (always false in M2)." Wire it: `db.progress(...)` joins `metamach_tasks` + `metamach_step_meta` and sets `tmux_alive = backend.has_session(session_id)`. This requires the daemon to hold a `DurableBackend` handle and pass session liveness into the progress query (or do a second pass). Minimal version: store the session name on the task/step row and let `progress` call `has_session`.

### 0.5 Tests (Phase 0)
- **Unit:** state-machine transitions (pure logic over a `FakeBackend`).
- **PG + tmux integration (ignored):** `utc_03_01b` — dispatch a 2-step local workflow (`test-flow`), assert `metamach_tasks` + `metamach_step_meta` transition `PENDING→STARTING→RUNNING→COMPLETED` and `Progress` returns the steps with `tmux_alive=true` while running. This is the currently-missing UTC-03-01b.

## Phase 1 — Cold-start re-exec (named Task 4.1 work)

### 1.1 Make `coldstart::reconcile` actually resume

Today `reconcile` logs "resumable task — pick up after `{step}`" with no re-exec. Change it to:

1. For each non-terminal task (`STARTING`/`RUNNING`/`SUSPENDED`):
   - Read the last `COMPLETED` step's `result_cache`/`stdout_tail` (the checkpoint).
   - Determine the **next step** to run (the step after the last `COMPLETED`; or the `current_step` if it was `RUNNING` and the tmux session is gone — re-run that step from its checkpoint).
   - Mint a fresh `tmux-janus-task-<task_id>-<new_uuid>` session (new UUID per ARCH §6.1 — never reuse a stale session name).
   - Call `workflow::resume_from(task_id, next_step)` — same as `run_workflow` but starting at a given step index, skipping already-`COMPLETED` steps.
2. Reject `tmux-resurrect` (already the case — cold start reads from PG checkpoints, never tmux state).

### 1.2 Resume semantics
- `resume_from` re-runs the workflow from `next_step` onward against the **same `task_id`** (preserves audit trail). It does NOT re-run completed steps.
- If the interrupted step was mid-`RUNNING` (no `COMPLETED` checkpoint for it), re-run that step (its `result_cache` is the last good state; the step is idempotent by contract or re-runnable). Mark `CONCURRENCY_RACE_ALERT` only if `target_sha` advanced (ties into Task 4.4 — see §3.2).

### 1.3 Tests (Phase 1)
- **PG + tmux integration (ignored):** `utc_03_03` (cold-start reconcile) — dispatch a 2-step workflow, kill the daemon mid-step (step 2 `RUNNING`), restart PG + daemon (NOT `make bootstrap`), assert `reconcile` resumes from the checkpoint and the task reaches `COMPLETED`. This is the currently-missing real UTC-03-03 (the existing test only checks `Ping` after restart).

## Phase 2 — Cross-host SSH transport (named Task 4.1 work)

### 2.1 `SshTmuxBackend` (`janus/src/tmux/ssh.rs`, new)

The `DurableBackend` trait already abstracts transport. Add a second impl that drives tmux on a **remote** host over SSH:

```rust
pub struct SshTmuxBackend { host: String, user: Option<String> }
impl DurableBackend for SshTmuxBackend { ... }  // each op shells: ssh [-l user] host tmux -L metamach-tmux ...
```

- `create_session`: `ssh <host> tmux -L metamach-tmux new-session -d -s <id> -c <cwd> '<command>'` with `remain-on-exit on`.
- `capture_pane` / `has_session` / `kill_session` / `list_sessions`: same `ssh ... tmux ...` prefix.
- The SSH binary is invoked via `std::process::Command` with `-o BatchMode=yes -o ConnectTimeout=5 -o StrictHostKeyChecking=accept-new` (matches the existing `ssh_probe`). No password handling — key-based only (air-gapped/dev-host model per Deployment-Spec).

### 2.2 Backend selection per step

A step declares `host` (`WorkflowStep.host`), falling back to the blueprint's `[remote]` (`recipe.remote_host`). The engine selects:
- `host` is `None`/`"local"` → `TmuxBackend` (local).
- `host` set → `SshTmuxBackend { host, user }`.

This is the per-step remote-override hook the recipe format already supports (`WorkflowStep.host`).

### 2.3 Tests (Phase 2)
- **Unit:** `SshTmuxBackend` command construction (assert the `ssh ... tmux ...` argv shape) against a stub — no real SSH.
- **SSH-gated (ignored, `requires SSH credentials`):** `utc_03_02` — dispatch a step whose `host` is a real SSH compile server, assert the session is created remotely and survives a packet-drop/re-attach. This is the deferred UTC-03-02; runs only with SSH creds (CI: `continue-on-error` step, per the PR #2 forward-compatible comment).

Not in this plan's scope but called out as the natural follow-on.

## 6. Risk / open questions for review

- **No absurd engine:** using MetaMach-owned task tables instead of absurd's `spawn_task`. Confirm this is acceptable (the absurd-integration assumption in the original M4 plan is dead — absurd was never added as a dep).
- **Exit-code capture from `remain-on-exit` tmux:** tmux keeps the pane after exit; reading the exit code requires ` PaneDead`/`pipe-pane` or parsing the pane. Need to confirm the capture mechanism (likely `capture-pane` + `$?` echo, or `display-message '#{pane_dead_status}'`). This is the fiddliest implementation detail.
- **janush-as-SHELL in dispatched sessions:** the step command runs under `janush` so Tool Guard intercepts. Confirm the dispatch should use `janush -c "<command>"` (matching the proxy-shell contract) vs. running the command directly. Recommend `janush` (interception is the point).
- **SSH transport security:** key-based only, `BatchMode=yes`. Confirm no password/agent-forwarding needs.
