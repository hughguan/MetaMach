# M4 Task 4.1 Remaining Work ‚Äî Design Plan (cross-host SSH + cold-start re-exec)

> **Status:** Design approved with adjustments (rev 2) - no implementation yet. Code lands in a later session.
> **Scope:** The last two-thirds of M4 Task 4.1: cross-host SSH transport + checkpoint-driven cold-start re-exec.
> **Rev 2 (review-driven):** commits to option (a) - `absurd.sql` never loaded, 002 migration comment fixed in Phase 0 (critical finding); adds `dispatch_sha` + `session_name` to `metamach_tasks` (findings #2/#4); repoints `cold_start_running_tasks` to the task table (#3); adopts `tmux display-message '#{pane_dead_status}'` for exit-code capture (#5); confirms `janush`-as-SHELL dispatch (#10).
> **Prerequisite finding:** Both are blocked on the **workflow-execution engine**, which does not exist. The daemon today handles only `Ping`/`Blueprints`/`Progress`/`GuardCheck`/`Onboard`/`Offboard` ‚Äî there is no `Dispatch`/`RunStep` request, no `spawn_task`, no step-state machine, and nothing drives a workflow's steps into `janus::tmux`. `janus::tmux` has session management (`DurableBackend`: create/attach/kill/has/capture) but **nothing calls it**. `coldstart::reconcile` scans for non-terminal tasks but finds none (nothing dispatches), so it only logs. `absurd` is branding (in-repo sqlx pool), not the external engine the comments reference.

This plan therefore covers **three phases**: the workflow-execution engine (Phase 0, the prerequisite), cold-start re-exec (Phase 1), and cross-host SSH transport (Phase 2). Phases 1 and 2 are the named Task 4.1 work; Phase 0 is the unblocker.

## Phase 0 ‚Äî Workflow-execution engine (prerequisite)

### 0.1 Schema: a task table

The per-blueprint DB today has only `metamach_step_meta` (per-step rows keyed by `task_id` + `step_name`). There is **no task table** ‚Äî a workflow run has no overall status row. The 002 comment assumed absurd's `absurd_tasks` table would provide this, but absurd isn't integrated.

**Decision:** add a MetaMach-owned `metamach_tasks` table (migration `004_tasks.sql`), applied alongside 002/003 in `ensure_blueprint_db`.

```sql
CREATE TABLE IF NOT EXISTS metamach_tasks (
    task_id          UUID PRIMARY KEY,
    blueprint_name   VARCHAR(100) NOT NULL,
    workflow_name    VARCHAR(100) NOT NULL,
    status           VARCHAR(20)  NOT NULL DEFAULT 'PENDING',  -- PENDING|STARTING|RUNNING|COMPLETED|FAILED|SUSPENDED
    current_step     VARCHAR(100),
    suspended_reason TEXT,
    dispatch_sha     VARCHAR(64),    -- Git HEAD at dispatch time (task-level; Task 4.4). All-zeros = non-git.
    session_name     VARCHAR(100),   -- tmux session id for this run (tmux_alive wiring; fresh UUID per resume)
    started_at       TIMESTAMPTZ,
    ended_at         TIMESTAMPTZ,
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_tasks_status ON metamach_tasks(status);
```

`metamach_step_meta` already carries per-step `status`/`exit_code`/`stdout_tail`/`started_at`/`hitl_verdict`/`target_sha` - reused as-is. **SHA fields split by level (review finding #2/#3):** `metamach_tasks.dispatch_sha` = HEAD at dispatch (task-level, for cold-start race detection); `metamach_step_meta.target_sha` = HEAD pinned for that step's execution (per-step optimistic lock). The task row is one row per run; `cold_start_running_tasks()` must be repointed to query `metamach_tasks` (one row per run) instead of aggregating `metamach_step_meta` (many rows per run) once Phase 0 lands.

### 0.2 Protocol: `Dispatch` contract

Add to `janus/src/protocol.rs`:

```rust
pub enum Request {
    // ...existing...
    Dispatch { blueprint: String, workflow: Option<String> },  // None => recipe.default_workflow
}

pub enum Response {
    // ...existing...
    Dispatch { task_id: Uuid },
}
```

Wire `Request::Dispatch` in the daemon handler (`janus_daemon.rs`) ‚Üí calls `workflow::run_workflow(...)`. Document as a new Contract (e.g. 3.9 Dispatch) in `Feature-Spec.md`.

### 0.3 The engine: `janus/src/workflow.rs` (new module)

`pub async fn run_workflow(db, backend: &dyn DurableBackend, recipe: &ValidatedRecipe, workflow: &Workflow, repo_root) -> Result<Uuid>`:

1. Insert a `metamach_tasks` row (status=`STARTING`), mint `task_id = Uuid::new_v4()`. Pin `dispatch_sha` = current git HEAD at `repo_root` (all-zeros sentinel for non-git blueprints; populates `metamach_tasks.dispatch_sha` for cold-start race detection - Task 4.4).
2. Resolve steps from the `Workflow` (`recipe::validate` already parsed `workflows/<default_workflow>.toml` into the `Workflow` struct during Onboard; re-load via `recipe::load_workflow`). For each step in order:
   - Upsert `metamach_step_meta` (status=`PENDING` -> `STARTING` -> `RUNNING`); set `task.current_step`. Pin `step.target_sha` = `task.dispatch_sha` (per-step optimistic lock).
   - Set task status=`RUNNING`.
   - Build the tmux session: `SessionId::new_for_task(task_id)`; the step runs under `janush` as `SHELL` (confirmed - review finding #10: Tool Guard interception is the point, so `janush` MUST be the entrypoint). The step's `command` field maps to `janush -c "<command>"` (i.e. `sh -c "<command>"` after the ALLOW verdict); set `JANUS_AGENT=<step.agent>` in the session env so the guard resolves the right agent profile. `cwd` = `repo_root`. Record the session name on `task.session_name`.
   - `backend.create_session(id, command, cwd)`.
   - Poll `backend` for session exit. **Exit-code capture (review finding #5):** use `tmux display-message -p -t <id> '#{pane_dead_status}'` - tmux-native, exact, and MetaMach already requires tmux 3.3+ (Deployment-Spec ¬ß1). `remain-on-exit` keeps the pane so the dead-status is readable. On exit: `backend.capture_pane(id)` -> `stdout_tail` (16 KiB budget via `truncate_16k`); set `step.exit_code`, `step.status` = `COMPLETED` (exit 0) | `FAILED` (non-zero).
   - On a Tool-Guard `BLOCK` (require_approval) verdict mid-step: set `step.status`=`SUSPENDED`, `step.hitl_verdict`=NULL (pending), `task.status`=`SUSPENDED`, `task.suspended_reason`, and **stop** (await HITL resume ‚Äî the resume/continue path is the M4 HITL loop, partly present via the gateway).
3. If all steps `COMPLETED`: `task.status`=`COMPLETED`, `ended_at=NOW()`. On any `FAILED`: `task.status`=`FAILED`.

**State machine** (matches the CLAUDE.md status enum):
- Task: `PENDING ‚Üí STARTING ‚Üí RUNNING ‚Üí COMPLETED | FAILED | SUSPENDED`
- Step: `PENDING ‚Üí STARTING ‚Üí RUNNING ‚Üí COMPLETED | FAILED | SUSPENDED`

### 0.4 `tmux_alive` wiring (Contract 3.3)

`ActiveTask.tmux_alive` is documented as "lands with Task 2.4 (always false in M2)." Wire it: `db.progress(...)` joins `metamach_tasks` + `metamach_step_meta` and sets `tmux_alive = backend.has_session(session_id)`. This requires the daemon to hold a `DurableBackend` handle and pass session liveness into the progress query (or do a second pass). The task row carries `session_name` (review finding #4); `progress` reads it and calls `backend.has_session(session_name)`.

### 0.5 Tests (Phase 0)
- **Unit:** state-machine transitions (pure logic over a `FakeBackend`).
- **PG + tmux integration (ignored):** `utc_03_01b` ‚Äî dispatch a 2-step local workflow (`test-flow`), assert `metamach_tasks` + `metamach_step_meta` transition `PENDING‚ÜíSTARTING‚ÜíRUNNING‚ÜíCOMPLETED` and `Progress` returns the steps with `tmux_alive=true` while running. This is the currently-missing UTC-03-01b.

## Phase 1 ‚Äî Cold-start re-exec (named Task 4.1 work)

### 1.1 Make `coldstart::reconcile` actually resume

Today `reconcile` logs "resumable task ‚Äî pick up after `{step}`" with no re-exec. Change it to:

1. For each non-terminal task (`STARTING`/`RUNNING`/`SUSPENDED`):
   - Read the last `COMPLETED` step's `result_cache`/`stdout_tail` (the checkpoint).
   - Determine the **next step** to run (the step after the last `COMPLETED`; or the `current_step` if it was `RUNNING` and the tmux session is gone ‚Äî re-run that step from its checkpoint).
   - Mint a fresh `tmux-janus-task-<task_id>-<new_uuid>` session (new UUID per ARCH ¬ß6.1 ‚Äî never reuse a stale session name).
   - Call `workflow::resume_from(task_id, next_step)` ‚Äî same as `run_workflow` but starting at a given step index, skipping already-`COMPLETED` steps.
2. Reject `tmux-resurrect` (already the case ‚Äî cold start reads from PG checkpoints, never tmux state).

### 1.2 Resume semantics
- `resume_from` re-runs the workflow from `next_step` onward against the **same `task_id`** (preserves audit trail). It does NOT re-run completed steps.
- If the interrupted step was mid-`RUNNING` (no `COMPLETED` checkpoint for it), re-run that step (its `result_cache` is the last good state; the step is idempotent by contract or re-runnable). Mark `CONCURRENCY_RACE_ALERT` only if `target_sha` advanced (ties into Task 4.4 ‚Äî see ¬ß3.2).

### 1.3 Tests (Phase 1)
- **PG + tmux integration (ignored):** `utc_03_03` (cold-start reconcile) ‚Äî dispatch a 2-step workflow, kill the daemon mid-step (step 2 `RUNNING`), restart PG + daemon (NOT `make bootstrap`), assert `reconcile` resumes from the checkpoint and the task reaches `COMPLETED`. This is the currently-missing real UTC-03-03 (the existing test only checks `Ping` after restart).

## Phase 2 ‚Äî Cross-host SSH transport (named Task 4.1 work)

### 2.1 `SshTmuxBackend` (`janus/src/tmux/ssh.rs`, new)

The `DurableBackend` trait already abstracts transport. Add a second impl that drives tmux on a **remote** host over SSH:

```rust
pub struct SshTmuxBackend { host: String, user: Option<String> }
impl DurableBackend for SshTmuxBackend { ... }  // each op shells: ssh [-l user] host tmux -L metamach-tmux ...
```

- `create_session`: `ssh <host> tmux -L metamach-tmux new-session -d -s <id> -c <cwd> '<command>'` with `remain-on-exit on`.
- `capture_pane` / `has_session` / `kill_session` / `list_sessions`: same `ssh ... tmux ...` prefix.
- The SSH binary is invoked via `std::process::Command` with `-o BatchMode=yes -o ConnectTimeout=5 -o StrictHostKeyChecking=accept-new` (matches the existing `ssh_probe`). No password handling ‚Äî key-based only (air-gapped/dev-host model per Deployment-Spec).

### 2.2 Backend selection per step

A step declares `host` (`WorkflowStep.host`), falling back to the blueprint's `[remote]` (`recipe.remote_host`). The engine selects:
- `host` is `None`/`"local"` ‚Üí `TmuxBackend` (local).
- `host` set ‚Üí `SshTmuxBackend { host, user }`.

This is the per-step remote-override hook the recipe format already supports (`WorkflowStep.host`).

### 2.3 Tests (Phase 2)
- **Unit:** `SshTmuxBackend` command construction (assert the `ssh ... tmux ...` argv shape) against a stub ‚Äî no real SSH.
- **SSH-gated (ignored, `requires SSH credentials`):** `utc_03_02` ‚Äî dispatch a step whose `host` is a real SSH compile server, assert the session is created remotely and survives a packet-drop/re-attach. This is the deferred UTC-03-02; runs only with SSH creds (CI: `continue-on-error` step, per the PR #2 forward-compatible comment).

### 3.2 Relationship to Task 4.4 (`target_sha` enforcement)
Task 4.4 (deferred) pins HEAD at two levels (review finding #2): `metamach_tasks.dispatch_sha` (HEAD at dispatch) and `metamach_step_meta.target_sha` (HEAD pinned per step). Phase 2's SSH transport is where stale reports arrive (slow remote compile over SSH). **Recommend implementing Task 4.4 alongside Phase 2** (or immediately after) ‚Äî the `target_sha` column already exists; Phase 2 adds the dispatch path that populates it. Not in this plan's scope but called out as the natural follow-on.
